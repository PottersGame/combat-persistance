## 2024-05-24 - Async Authentication Race Condition & Blocking IO
**Vulnerability:** The `AuthManager` performs blocking `BCrypt` hashing inside `CompletableFuture.runAsync()`, which uses the default `ForkJoinPool`. Even worse, it updates the `authenticatedPlayers` set and `users` map within the async task, before verifying if the player is still connected on the main thread in `AuthCommands`. This leads to a race condition where a player can disconnect mid-authentication but still be marked as authenticated in the in-memory map.
**Learning:** Security state transitions (like logging in) must be atomic and synchronized with the game loop. Additionally, computationally heavy operations like `BCrypt` can starve the default async executor, causing delays for other tasks.
**Prevention:** Perform blocking hashes on a dedicated `IO_EXECUTOR`. Return the result to the main thread, verify the player is still connected using a reliable check (like `player.hasDisconnected()`), and only then apply state changes synchronously.
